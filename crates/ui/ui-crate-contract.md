# Архитектурный контракт для виджетов `egui-android-ui`

Это не просто набор правил — это жёсткие гарантии, которые должен соблюдать каждый виджет,
модификатор и контейнер, чтобы API оставался Compose-like, предсказуемым, расширяемым и безопасным.

Контракт написан так, чтобы его можно было использовать:
- как основу для ревью PR,
- как руководство для разработчиков,
- как спецификацию для автоматического рефакторинга,
- как входной документ для ИИ-агента.

---

## 1. Общие принципы виджетов

Каждый виджет в `egui-android-ui` обязан соблюдать следующие фундаментальные правила:

### 1.1 Виджет — чистая декларация

Виджет не содержит бизнес-логики, не читает State, не вызывает Store, не знает про Android.

Он только:
- описывает структуру UI,
- измеряет свой контент,
- рисует себя,
- диспатчит сообщения через `Dispatcher<M>`.

### 1.2 Виджет не имеет побочных эффектов

Метод `render(ui, dispatch)`:
- не должен изменять глобальное состояние,
- не должен выполнять async-операции,
- не должен вызывать внешние сервисы,
- не должен изменять State напрямую.

### 1.3 Виджет полностью управляется модификаторами

Все изменения поведения (размер, фон, кликабельность, выравнивание, анимации)
должны происходить через модификаторы.

Виджет не должен иметь собственные поля, дублирующие функциональность модификаторов.

---

## 2. Layout-pipeline (обязательный)

Каждый виджет обязан реализовывать трёхфазный layout, аналогичный Jetpack Compose:

### 2.1 Measure phase

Виджет обязан:
- измерить свой контент,
- получить точный размер (`content_size`),
- учесть модификаторы размера (`Modifier.size`, `fill_max_width`, `fill_max_height`),
- учесть padding/background/align.

### 2.2 Layout phase

Виджет обязан:
- вызвать `ui.allocate_exact_size(content_size, sense)` или `ui.allocate_space(content_size)`
  в зависимости от интерактивности,
- **не использовать `available_size()` как размер по умолчанию**.

### 2.3 Paint phase

Виджет обязан:
- рисовать контент строго внутри выделенной области,
- использовать `ui.painter()` только после layout.

---

## 3. Правила для модификаторов

### 3.1 Модификатор — чистая функция над layout

Модификатор обязан:
- не изменять State,
- не диспатчить сообщения,
- не выполнять side-effects,
- не менять поведение других модификаторов.

### 3.2 Модификаторы не конфликтуют друг с другом

Комбинации:
- `padding + background`
- `size + clickable`
- `align + background`
- `padding + clickable + size`

должны работать детерминированно.

### 3.3 Модификаторы применяются в строгом порядке

Порядок:
1. size
2. padding
3. background
4. align
5. clickable
6. animation

### 3.4 Каждый модификатор рендерит контент ровно один раз

Запрещён двойной рендер: сначала в `child_ui` для измерения, потом повторно.
Аллоцированная область должна совпадать с отрендеренной.

---

## 4. Правила для Clickable / ClickableWith

### 4.1 Размер клика — размер контента

Clickable обязан:
- измерить контент,
- использовать `content_size`,
- **не использовать `available_size()`**.

### 4.2 Никакого двойного рендера

Clickable обязан:
- рендерить контент **только один раз**,
- не создавать `child_ui`, который не рисует в родительский painter.

### 4.3 Clickable не меняет layout контейнера

Clickable не должен:
- растягивать Row/Column,
- менять min_size контейнера,
- влиять на scroll-расчёт LazyColumn.

### 4.4 Response должен соответствовать отрендеренной области

clickable-область должна визуально и логически совпадать с alloc'нутой областью.

---

## 5. Правила для Button

### 5.1 Wrap-content по умолчанию

Кнопка обязана:
- измерять текст через galley,
- учитывать внутренний padding (по умолчанию `12.0px` горизонтальный, `8.0px` вертикальный),
- **не растягиваться по ширине без модификатора**.

### 5.2 Full-width только через модификатор

Только:
- `.size(...)`,
- `.fill_max_width()`,
- `Modifier::new().fill_max_width()`

может растянуть кнопку.

### 5.3 Многострочный текст обязателен

Button обязан:
- использовать multiline layout,
- учитывать `ui.available_width()` если задан `Modifier::fill_max_width()`.

---

## 6. Правила для Text

### 6.1 Единое поведение selectable/non-selectable

Оба режима обязаны:
- использовать wrap-content,
- учитывать модификаторы,
- иметь **одинаковый layout-pipeline** (measure → allocate → paint).

### 6.2 Текст не должен растягивать контейнеры

Text обязан:
- использовать wrap (перенос строк),
- учитывать `max_width` контейнера,
- не выходить за границы родителя.

---

## 7. Правила для Spacer

### 7.1 Spacer — всегда allocate_exact_size

Spacer обязан:
- не использовать `add_space`,
- не использовать `available_size()`.

### 7.2 Spacer не должен ломать Row/Column

Spacer обязан:
- корректно работать с модификаторами,
- не менять направление layout,
- иметь корректный Sense (hover, не click).

---

## 8. Правила для Stack

### 8.1 Stack — wrap-content

Stack обязан:
- измерить **всех** детей,
- взять `max(child_size)` по каждой оси,
- аллоцировать ровно этот размер,
- **не использовать `ui.available_size()`**.

### 8.2 Stack не должен занимать весь экран

Stack **не должен** использовать:
```rust
ui.available_size()
```

### 8.3 Каждый ребёнок Stack имеет уникальный id

При рендере каждого ребёнка Stack обязан использовать `ui.push_id(index)` или
уникальный `id_salt`, чтобы `remember()` работал корректно.

---

## 9. Правила для анимаций

### 9.1 Анимации — часть Modifier системы

Анимации обязаны:
- быть узлами в `ModifierNode` value type,
- быть частью Modifier chain,
- не менять layout,
- не менять размер виджета.

---

## 10. Правила для контейнеров (Column, Row)

### 10.1 Spacing консистентен

По умолчанию spacing = `8.0` для всех контейнеров.

### 10.2 Контейнеры не растягивают детей

Контейнер обязан:
- не принуждать детей растягиваться на всю доступную ширину/высоту,
- уважать wrap-content размер каждого ребёнка.

---

## 11. Анимационные модификаторы (ModifierNode)

### 11.1 Fade как модификатор

`Modifier::new().fade(opacity)` должен создавать узел `ModifierNode::Fade(f32)`,
который оборачивает контент в `ui.scope()` с `multiply_opacity()`.

### 11.2 Slide как модификатор

`Modifier::new().slide(direction, offset)` должен создавать узел `ModifierNode::Slide`,
который оборачивает контент со смещением.

---

## 12. Правила для тестов

Каждый виджет обязан иметь тесты:
- wrap-content,
- модификаторы,
- clickable,
- nested containers,
- LazyColumn,
- Stack,
- multiline text,
- negative modifiers (ошибки).

---

## 13. Порядок применения модификаторов в `apply_recursive`

`Modifier::apply()` применяет модификаторы **в порядке добавления** (первый — самый внешний):

```
Modifier::new()
    .fill_max_width()       // 1. Резервирует всю ширину
    .padding(16.0)          // 2. Внутри — отступ
    .background(RED)        // 3. Фон рисуется поверх padding
    .clickable(Msg::Click)  // 4. Кликабельная область = размер контента
```

**Порядок имеет значение:**
- `.fill_max_width().padding(16.0)` — padding внутри полной ширины
- `.padding(16.0).fill_max_width()` — сначала padding, потом растяжение (может не дать эффекта)
- `.background(RED).padding(16.0)` — padding снаружи фона
- `.padding(16.0).background(RED)` — фон поверх padding

---

## 14. Итог

Этот документ — жёсткая спецификация, которую должен соблюдать каждый виджет,
модификатор и контейнер.

Если что-то нарушает контракт — это архитектурная ошибка.
