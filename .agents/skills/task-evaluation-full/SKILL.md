## Шаг 1. Глобальная оценка — ЗАЧЕМ?

**С точки зрения пользователя продукта** (конечного пользователя Android-приложения):
- Какую проблему пользователя мы решаем?
- Что он должен увидеть / почувствовать / получить в итоге?

**С точки зрения разработчика** (того, кто использует фреймворк):
- Какую боль разработчика мы снимаем?
- Что станет проще / понятнее / надёжнее?

> ⚠️ Если не можешь чётко сформулировать оба ответа — **остановись и уточни у пользователя**.

---

## Шаг 2. Локальная оценка — ЧТО?

Сформулируй задачу в **одном предложении**: «Задача: <что конкретно должно быть сделано>»

**Критерии успеха** (3–5 пунктов):
- [ ] Критерий 1
- [ ] Критерий 2
- [ ] ...

**Границы задачи** (что НЕ делаем):
- ❌ Не делаем X
- ❌ Не трогаем Y

---

## Шаг 3. План с приоритетом инвазивности

Составь план из 3–7 шагов. **Порядок: от наиболее инвазивных к наименее инвазивным.**

Уровни инвазивности:
- 🔴 **Высокий** — изменение публичного API / сигнатуры, затрагивает 5+ мест
- 🟠 **Средний** — изменение внутреннего поведения с сохранением API
- 🟡 **Низкий** — локальный фикс в одном файле
- 🟢 **Косметика** — комментарии, форматирование, логи

**Правило:** инвазивные изменения идут первыми — они каскадно затрагивают связанные участки. Локальные фиксы, сделанные до них, часто приходится переписывать.

**Быстрая проверка инвазивности** (если ≥2 из 3 — делаем первым):
1. Затрагивает >3 файлов?
2. Меняет публичный API?
3. Ломает тесты/примеры?

> 🚩 Если в середине плана обнаружилось инвазивное изменение — **остановись, переупорядочи план, покажи пользователю**.

---

## Шаг 4. Покрытие тестами

Для каждого шага плана определи, **что и как тестировать**:

**Юнит-тесты** (изолированная логика):
- Чистые функции (reducer'ы, утилиты, парсеры)
- Отдельные методы с моками зависимостей
- Граничные случаи и ошибки

**Интеграционные тесты** (взаимодействие компонентов):
- Взаимодействие Component ↔ StateStore ↔ Dispatcher
- ChildStack push/pop с lifecycle-событиями
- Полный MVI-цикл: View → dispatch → handle → state → render

**Моки и заглушки** (где надо):
- `egui::Context` — через `__run_test_ui` или `Context::default()`
- Каналы (`mpsc`) — реальные каналы в тестах, не моки
- Платформа — мок `AndroidBackend` для lifecycle/input
- Внешние зависимости — trait-объекты с тестовыми реализациями

**Критерий покрытия:** каждый публичный метод и каждая ветка логики должна иметь ≥1 тест. Если шаг не покрывается тестами — укажи почему.

**Критерии успеха:**
- [ ] ...

**Границы (не делаем):**

**План:**
| # | Шаг | Инвазивность | Тесты |

## Красные флаги (стоп → вопрос пользователю)

- 🚩 Хочется переделать что-то «заодно»
- 🚩 Появилась идея «лучшего» решения, не обсуждённого с пользователем
- 🚩 Задача разрастается за пределы границ
- 🚩 Неясно, как проверить результат или покрыть тестами
